## Root Cause Classification

### Primary: NaN Comparison Bug (7/8 failures)
- **Script**: `common/correctness/correctness.py:check_correctness` at line 128
- `torch.allclose` is called with `equal_nan=False` (default)
- NaN has multiple bitwise representations in IEEE 754 FP16
- Torch NPU produces one NaN encoding, numpy produces another
- NaN positions are identical (NaN mask matches perfectly)
- **Fix**: Change `equal_nan=False` to `equal_nan=True` in correctness checker

### Secondary: Stale Reference Data (1/8 failure — B=64 random_finite)
- The `correctness_results.json` is stale (Jul 16)
- Current reference data was regenerated (Jul 17)
- With current data, the test passes (max_abs=0.001953125 < atol=0.01)

### Tertiary: Data Generation Bugs (2x pos_neg_cancel, 2x nan inputs)
- `x_b1_pos_neg_cancel.bin` and `x_b16_pos_neg_cancel.bin` are 2x expected size
- `x_b1_nan.bin` and `x_b16_nan.bin` have only 1 NaN instead of all NaN
- Likely from a corrupted regeneration of input data
- These are NOT counted in the 8 FAIL results since they would cause script crash, not FAIL status
- Root cause suspected: `generate_inputs.py` seed collision or file overwrite race condition
