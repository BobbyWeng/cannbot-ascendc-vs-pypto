# 流程重构：设计摘要（中文版）

## 概述

流程重构将原始 AGENTS.md（松散地描述三条路线的内联技能调用）替换为一个结构化的治理框架，基于：

1. **强制任务分类** — 每个任务在开始前沿 4 个轴进行分类。
2. **基于插件的路由** — 正确的插件（ops-direct-invoke、pypto-op-orchestrator 等）作为整体加载，而不是分散的技能。
3. **15 门禁生命周期** — 顺序门禁，明确的通过/失败标准，且正确性门禁**绝不可覆盖**。
4. **状态持久化** — 每个算子获得遵循共享模式的 TASK_STATE.json。
5. **Cannbot 使用验证** — 自动化检查确保框架确实被使用。
6. **强制脚本** — 提交前和发布前门禁自动运行。

## 关键设计决策

### 为什么不是单一编排器？
项目比较三个后端（Torch、Ascend C、PyPTO）——它们有根本不同的工作流。强行将它们放入一个编排器要么太抽象而无用，要么太复杂而难以维护。相反，每条路线使用其现有插件的工作流，项目级框架在其周围添加治理门禁。

### 为什么是 15 个门禁？
每个门禁对应一个不同的生命周期阶段。明确的门禁意味着：
- 每个阶段有明确的通过/失败
- 没有跳过的阶段
- G8（正确性）明确声明绝不可覆盖——即使通过覆盖机制也不行
- G13（代码审查）可以覆盖，但必须记录覆盖原因，结果仅为 DRAFT

### 为什么除了编排器状态还需要 TASK_STATE.json？
`.orchestrator_state.json` 是 PyPTO 特定的。项目级 `TASK_STATE.json` 统一覆盖所有后端，并遵循共享的 JSON Schema。
