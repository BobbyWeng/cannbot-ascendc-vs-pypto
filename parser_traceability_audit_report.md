# Parser Traceability Audit Report

## Executive Summary

**Date**: 2026-07-17
**Scope**: `/mnt/workspace/cannbot_ascendc_vs_pypto` — parser code, parsed data files (11 operators), raw profiler data, benchmark scripts, final reports
**Method**: CPU-only file audit, cross-referencing parser code → parsed JSON → final reports

**Finding**: The traceability chain is **broken**. The parser (`parse_profiler.py`) has multiple bugs and data gaps. The final reports for `mul` and `relu` contain values that **cannot be traced back** to any field in the current parsed JSON. PyPTO's `primary_compute_kernel_us` is systematically conflated with `per_call_us` in all add and mul final reports. Four operators (`not`, `or`, `equal`, `where`) have NO parsed data at all — only raw host-sync Event timestamps.

---

## Step 1: Parser Audit (`parse_profiler.py`)

### Bug 1: `primary_compute_kernel_us` = max single kernel, NOT per-call

**Line 61**: `primary = max(compute_kernels, key=lambda e: e['dur_us'])`

This selects the **longest single kernel event** across ALL iterations (100 loops × N kernels/loop). It does NOT divide by `loops`. This is correct for a "worst-case single kernel" metric but is **confusingly labeled** — users expect `primary_compute_kernel_us` to mean "per logical call."

In the add report, torch/ascendc correctly use this max-single-kernel value. PyPTO uses `per_call_us` (mean of all non-AICPU kernels per call). This is an **inconsistency** across implementations in the same report.

**BUT** in the mul and relu final reports, the numbers match **NEITHER** `primary_compute_kernel_us` nor `per_call_us` — they were generated by a different (earlier) parser.

### Bug 2: No per-call primary compute kernel metric

The parser computes `kernel_type_breakdown[X]['per_call_us']` = `total_us / loops`, but this is the **sum** of all kernel events of that type per call, not the primary compute kernel duration per call. There is NO field that gives `primary_compute_kernel_us / loops` (the average max kernel event per call).

### Bug 3: No raw statistics (min, median, P90, std, CV)

The parser extracts only `mean_us` per kernel type. It does **not** compute:
- Per-iteration kernel durations (the trace has 100×N events, but all are aggregated)
- Minimum / median / P90 / standard deviation / coefficient of variation
- These statistics are **only** available from the host-side benchmark scripts (Event-based), not from msprof trace parsing

### Bug 4: `kernels_per_logical_call` = total_kernels / loops (naive)

**Line 69**: `kernels_per_call = round(len(kernel_events) / loops, 2)`

This assumes every logical call produces the same number of kernel events. For simple operators (add, mul, relu with 7 events per iteration) this works. For PyPTO ops with AICPU executors, the AICPU events are included in this count (e.g., 201 AICPU + 100 MIX_AIC = 301 total for 100 loops = 3.01 per call). This is **correct but misleading** — `kernels_per_logical_call` includes AICPU executor events that are NOT compute kernels.

### Bug 5: AICPU misidentification risk

**Lines 59-63**: The parser filters AICPU kernel events out via `'AICPU' not in e['type']`. This is a **substring check** on the Task Type field. If a future Ascend C kernel contains "AICPU" in its name, it would be misclassified. Currently safe (KERNEL_AICPU is the only type matching), but fragile.

### Bug 6: AICPU primary compute kernel fallback

**Line 62-63**: `if compute_kernels: primary = max(compute_kernels, ...) else: primary = max(kernel_events, ...)`. If ALL kernel events are AICPU (no non-AICPU), the parser falls back to selecting the max from ALL events. This is reasonable but would silently select an AICPU executor as the "primary compute kernel."

### Bug 7: No msprof raw directory hash or provenance

The parser does **not** record:
- Which msprof raw directory was parsed (PROF_* hash)
- `source_hash` or `binary_hash` of the kernel binary
- Timestamp of the raw msprof data
- md5/sha256 of the raw msprof JSON file

### Bug 8: `--loops` parameter reliability

**Line 111**: Default `loops=100`. The parser trusts the caller to provide the correct value. If the caller passes `--loops 100` but the actual profiled iterations were 50, `per_call_us` and `kernels_per_logical_call` would be **off by 2x**.

### Bug 9: Multi-kernel scenario handling

The parser groups by `Task Type` (e.g., KERNEL_AIVEC, KERNEL_AICORE, KERNEL_MIX_AIC) and reports breakdown per type. If an operator uses 3 different kernel types, each gets its own breakdown. **However**, `primary_compute_kernel_us` only reports the single max event — not the per-type max. A multi-kernel operator where the second kernel type dominates in total time would be poorly represented.

### Bug 10: `primary_compute_kernel_us` for PyPTO includes AICPU when it's the only type

In `expand` pypto_b32.json, `primary_compute_type` is `KERNEL_AICPU` (value 2918.7us) — the AICPU executor dominates and becomes the "primary compute kernel." This violates the project's stated goal of excluding AICPU from compute comparisons.

### Bug 11: `host_events_count` only exists in old-format parsed files

The `aggregate.json` files (expand, transpose) contain `host_events_count` but the current parser does NOT output it. The old parser version collected host API event counts; the current one does not.

## Step 3: Known Parser Bug Verification

### Bug A: PyPTO primary selected longest AICPU executor

**CONFIRMED**: In `expand` pypto_b32.json and aggregate.json, `primary_compute_type` = `KERNEL_AICPU` (2918.7us). The AICPU executor is the longest event because PYPTO_expand_row (KERNEL_MIX_AIC) is shorter per event. Since MIX_AIC is not filtered by `'AICPU' not in e['type']`, the MIX_AIC events are eligible as compute kernels. But the AICPU events' max (2918.7) > MIX_AIC max, so AICPU wins. **This is WRONG** because AICPU executors should never be the primary compute kernel.

**Fixed in current parser?** Partially — current parser (lines 59-63) filters AICPU out of compute_kernels. But `expand pypto_b32.json` was parsed with an OLD parser that didn't do this filtering.

### Bug B: `mean_us` vs `primary_compute_kernel_us` confusion

**CONFIRMED**: In the ADD final report, PyPTO's `primary_compute_kernel_us` field actually stores `per_call_us` (mean of all KERNEL_MIX_AIC events per call), NOT `primary_compute_kernel_us` (max single KERNEL_MIX_AIC event). For torch/ascendc, the same field stores the actual `primary_compute_kernel_us`.

Example: add/pypto b1 — `primary_compute_kernel_us` in parsed = 132.123us (max single kernel) but final report = 136.03us (`per_call_us` = sum of all KERNEL_MIX_AIC / 100 loops). These differ by ~3%.

### Bug C: Single kernel average vs logical call confusion

**CONFIRMED**: All seven non-AICPU operators (add, mul, relu, div, etc.) have `kernels_per_logical_call` > 1 (typically 7). The final reports sometimes divide by this to get "average single kernel duration." This is correct when all kernels have similar duration, but wrong for tiled kernels with wide variance.

### Bug D: Raw data existing but not parsed

**CONFIRMED**: Four operators (not, or, equal, where) have `reports/raw/{impl}/b{*}/result.json` files but NO `reports/parsed/` directory at all. The raw msprof PROF_* directories are also absent (these were Event-timed only).

### Bug E: Same batch reading wrong directory

**OBSERVED**: In `expand`, there is a `ascendc_b32FIX` directory alongside `ascendc_b32`. Both have parsed JSON. The `ascendc_b32.json` contains `"error": "no kernel events found"` while `ascendc_b32FIX.json` has valid data. This indicates the first profiling attempt failed (no kernel events) and was re-run.

## Step 4: Benchmark Script Audit

### Torch benchmark scripts

The torch benchmark scripts (`operators/{op}/torch/benchmark.py`) use `torch.npu.Event` for timing. They:
- Do NOT invoke `msprof` directly
- Do NOT call `parse_profiler.py`
- Save results to `benchmark_results.json` (NOT to `reports/raw/`)
- Use different shapes than the unified pipeline

The unified profiler (`common/profiler/run_profiler.py`) handles torch differently:
- Uses inline-generated scripts that invoke msprof for torch ops (for msprof-capable ops)
- Uses Event-based scripts for not, or, equal, where

### Ascend C benchmark scripts

No standalone benchmark scripts exist per operator. The Ascend C profiler is invoked via `run_profiler.py` which:
- Runs the compiled Ascend C binary direct
- Extracts RESULT lines from stdout to create `result.json`
- The binary itself does the timing and outputs to stdout
- `run_profiler.py` does NOT call `parse_profiler.py`

### PyPTO benchmark scripts

- `add/pypto/benchmark.py`: Event-based, NO msprof integration
- `not` and `or` PyPTO: Inline scripts in `run_profiler.py`, Event-based
- Other PyPTO ops (mul, relu): Not explicitly profiled in benchmark dirs; profiler data comes from run_profiler.py

### Profiler integration gaps

1. **No script calls `parse_profiler.py`** — parsing was done manually or by a non-reproducible process
2. **No `run_all.sh`** exists in any operator's benchmark directory
3. **Path inconsistency**: Torch benchmark scripts save to `torch/benchmark_results.json` while the raw profiler data lives in `reports/raw/{impl}_b{N}/`. These are separate measurement pipelines.
4. **The `run_profiler.py` script** only works for operators with a `run_ascendc_profiler` path (has compiled binary) and `run_pypto_profiler` (only supports `not` and `or`)

## Summary of Findings

### Critical (blocking traceability)

1. **Final reports for relu and mul are UNTRACEABLE** — the values (e.g., torch B1 = 2.635us for relu) do not match `primary_compute_kernel_us` (10.08), `mean_us` (2.622), or `per_call_us` (18.352) from any current parsed file. These reports were generated with a different (unknown) parser.

2. **Four operators (not, or, equal, where) have NO parsed data** — they have Event-based host timing only. Their final reports are NOT comparable with msprof-based operators.

3. **No source/binary hashes in ANY parsed file** — zero traceability from parsed data back to the kernel binary that was profiled.

4. **Parser `--loops` parameter is unchecked** — if the wrong value is passed, all `_per_call` metrics are wrong.

### High (accuracy impact)

5. **PyPTO's `primary_compute_kernel_us` in add final report is actually `per_call_us`** (e.g., 136.03 instead of 132.123). The difference is ~3% for add, but the concept is wrong — `primary_compute_kernel_us` should be the max single kernel event per call, not the sum divided by loops.

6. **All `primary_compute_kernel_us` values are max single kernel across ALL iterations** — not the average max per call. This inflates the value by including one outlier iteration with a long kernel.

7. **In expand pypto, the old parser selected AICPU executor as primary** — 2918.7us (KERNEL_AICPU) vs the actual compute kernel which is ~46us (KERNEL_MIX_AIC).

### Medium (completeness)

8. **No per-iteration statistics** in parsed data (min, median, P90, std, CV).
9. **No host API synced time** — the parser collects but does NOT output host API events (`host_api_events` is counted but discarded).
10. **Path convention inconsistency** — `not`/`equal`/`where`/`or` use `reports/raw/{impl}/b{*}/result.json` while arithmetic ops use `reports/raw/{impl}_b{*}/` with PROF_ directories.

### Recommendations

1. **Add `source_hash` and `raw_msprof_dir`** to the parser output.
2. **Add per-iteration statistics** (min, median, P90, std, CV) to the kernel_type_breakdown.
3. **Add `primary_compute_kernel_per_call_us`** = max kernel duration per logical call, averaged across iterations.
4. **Fix the `--loops` parameter** to auto-detect from the trace if possible, or validate against kernel_count.
5. **Re-parse all operators** with the fixed parser to ensure traceability.
6. **Regenerate final reports** for relu and mul after re-parsing.
7. **Add parsed data files** for not, or, equal, where operators (even if Event-based, they should be documented as `host_sync` level).
8. **Tag all parsed JSON with `parser_version`** string.
